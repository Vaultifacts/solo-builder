# Solo Builder Workflow Specification

## Purpose
This document is the canonical written specification for the deterministic Solo Builder workflow as currently practiced in this repository. It documents current behavior only and does not define new semantics.

## Scope and source-of-truth model
- Machine state source of truth: `claude/STATE.json` (local runtime state, gitignored).
- Agent-facing rendered contract: `claude/NEXT_ACTION.md` (generated by orchestrator, gitignored).
- Task planning/history artifacts: `claude/TASK_QUEUE.md`, `claude/TASK_ACTIVE.md`, `claude/JOURNAL.md`.
- Enforcement scripts live under `tools/`.

## Workflow phases and roles
Script-enforced phases:
- `triage`
- `research`
- `plan`
- `build`
- `verify`
- `done`

Script-enforced roles:
- `RESEARCH`
- `ARCHITECT`
- `DEV`
- `AUDITOR`

Script-enforced non-done phase mapping:
- `triage -> RESEARCH`
- `research -> ARCHITECT`
- `plan -> DEV`
- `build -> DEV`
- `verify -> AUDITOR`

Done-phase behavior:
- `phase=done` is terminal for the task.
- Orchestrator output must respect persisted `next_role` when rendering done-phase prompts/contracts.

## Task lifecycle
### Script-enforced lifecycle operations
1. Role-specific work is produced in `claude/HANDOFF_*.md` artifacts.
2. State transitions are applied via `pwsh tools/advance_state.ps1`.
3. Next-role execution contract is rendered via `pwsh tools/claude_orchestrate.ps1` into `claude/NEXT_ACTION.md`.
4. Verification runs via `pwsh tools/audit_check.ps1` and writes `claude/verify_last.json`.
5. `dev_gate`/pre-commit checks enforce guarded development behavior.

### Convention-driven lifecycle discipline
1. Initialize task metadata and commit task-init changes.
2. Execute deterministic role flow: `RESEARCH -> ARCHITECT -> DEV -> AUDITOR`.
3. Close task after audit PASS and state transition to `done`.
4. Record durable closeout journal history as a separate closeout commit when appropriate.

## Branch lifecycle
### Convention-driven branch rule (merge-first baseline)
- New task branches must be created from updated `master` (or agreed integration branch), never from a prior task branch.

### Canonical scripted initialization path
- Use `pwsh tools/start_task.ps1 -TaskId TASK-XXX -Goal "<goal>"` as the standard initializer.
- The script enforces preflight gating automatically:
  - switch to `master` (and optional pull when upstream exists)
  - merge prior task branch when applicable
  - run `pwsh tools/workflow_preflight.ps1`
  - abort on nonzero preflight
  - create `task/TASK-XXX` only after preflight passes

Recommended sequence:
1. `git checkout master`
2. `git pull` (if upstream is configured)
3. Merge completed prior task branch if not already integrated:
   - `git merge --no-ff task/TASK-XXX`
4. Confirm clean baseline on `master`.
5. Create next task branch:
   - `git checkout -b task/TASK-YYY`

## Closeout procedure
### Script-enforced closeout pieces
- Auditor verification through `pwsh tools/audit_check.ps1`.
- State transition to done via `pwsh tools/advance_state.ps1 -ToPhase done ...`.
- Orchestrator renders post-close contract via `pwsh tools/claude_orchestrate.ps1`.

### Convention-driven closeout pieces
1. Commit audit result (`claude/HANDOFF_AUDIT.md`).
2. Restore local runtime artifacts (see next section).
3. Inspect `claude/JOURNAL.md` and:
   - commit if it contains durable closeout history
   - restore if it is only runtime noise
4. Merge completed task branch to `master` before creating next task branch.

## Local-only runtime artifact handling
Runtime/local artifacts are typically gitignored and should not be included in normal task commits unless explicitly required by workflow policy.

Common local runtime artifacts:
- `claude/STATE.json`
- `claude/NEXT_ACTION.md`
- `claude/allowed_files.txt`
- `claude/verify_last.json`
- `claude/logs/*`
- `claude/snapshots/*`

## CI verification-only contract
- CI must run verification-only workflow checks and must not advance lifecycle state.
- Canonical CI invariant command path: `pwsh tools/ci_invariant_check.ps1`.
- CI must not run:
  - `tools/advance_state.ps1`
  - `tools/start_task.ps1`
  - branch creation/switch/merge operations
- CI checks are expected to validate:
  - `STATE.json` <-> `NEXT_ACTION.md` consistency
  - required verification commands from `claude/VERIFY.json`

Typical cleanup:
- `git restore --source=HEAD --worktree --staged claude/allowed_files.txt claude/verify_last.json`
- Inspect `claude/JOURNAL.md` before deciding restore vs commit.

## Enforced vs convention summary
Script-enforced:
- state transitions, phase/role validation, and orchestration rendering
- verification contract execution and verify output emission
- development guard checks and pre-commit gate behavior

Convention-driven:
- merge-first baseline discipline
- task branch/init sequencing conventions
- closeout cleanup cadence for runtime artifacts
- commit granularity/message discipline across role stages

## Non-goals of this specification
- No changes to workflow semantics.
- No changes to orchestrator/state-machine logic.
- No changes to product behavior under `solo_builder/*`.
